04 · 源码LESSON 18 50–70 分钟分布式入门

这一课要回答：actor、rollout、critic、Ray actor、rank 和 GPU 到底怎样对应？

Worker 与资源编排：谁占 GPU，谁只是控制器 ​

veRL 用 Ray 把逻辑角色放进资源池，再由 WorkerGroup 统一分发调用。读这部分时分开三个概念：角色说明职责，资源池说明放置，模型引擎说明如何并行计算。

先用人话：岗位、工位与工作方式 ​

  • role 是岗位：actor 负责更新策略，rollout 负责生成；
  • ResourcePool 是工位分配：哪些节点/GPU 可被这个岗位占用；
  • WorkerGroup 是同一岗位的多 rank 组织与调用入口；
  • FSDP/Megatron/VeOmni 等 model engine 是岗位内部怎样拆分计算；
  • Ray actor 是运行时进程抽象，不等于 RL 的 actor policy。

把这五层混起来，就会出现“我有 8 张 GPU，为什么 actor 还 pending”这类无法回答的问题。

角色关系 ​

同一逻辑 role 可以根据 FSDP、Megatron、VeOmni 等 model_engine 选择不同 worker class。反过来，多个 role 也可能共置在一个 Ray actor/资源池中，以便共享 GPU，但它们的生命周期方法仍不同。

核心角色 ​

角色主要工作何时可省略
Actor重算 log-prob、执行 policy loss、更新参数不能省略
Rollout通过 LLM server 生成 trajectory不能省略，但可与 actor 共置
Reference为 KL 提供冻结概率没有 reward KL/actor KL 时可不启用
Critic预测 value、训练 value lossGRPO/RLOO 等无 critic 配方可省略
RewardModel模型式打分纯规则奖励可省略独立 RM
Teacheron-policy distillation logits/信号非蒸馏实验省略

need_reference_policy() 与 need_critic() 才是最终角色判断入口，不要只看某个孤立的 enable 字段。

ResourcePool 与 WorkerGroup ​

ResourcePoolManager 建立角色到 pool 的映射；create_colocated_worker_cls 可以把同一 pool 中的多个 worker class 包装成共置 actor；RayWorkerGroup.spawn() 再建立各 rank。调用 compute_log_prob、update_actor 等方法时，dispatch 层会按 DP mapping 切分 KVBatchMeta，worker 通过 TQ 读取字段。

因此 batch 约束至少涉及：

  • 轨迹条数能否被 data-parallel size 合理切分；
  • actor/critic 的 global mini-batch 与 rollout.n；
  • sequence-length balance 后每个 rank 的 token workload；
  • tensor/pipeline/expert parallel 对单副本 GPU 数的要求。

单机与多机的实际边界 ​

Ray 负责调度，但不会自动让所有路径和网络接口正确：

  • 模型、数据、checkpoint 与输出目录必须在对应 worker 节点可访问；
  • 每节点对外通信 IP 要正确，不能让 collective 选到 loopback；
  • rollout server 的 host IP 通常是节点局部值，不能一份固定环境变量复制到所有节点；
  • NCCL/Gloo/设备可见性环境应在 Ray 节点/actor 启动前生效；
  • /tmp、Ray session、模型 cache 与 TQ storage 要有足够容量。

资源问题的最小证据集 ​

遇到 pending actor、OOM 或 collective timeout 时，保存：

bash
ray status
nvidia-smi                 # Ascend 环境换成对应工具
df -h /tmp "$HOME"
python -c 'import ray, verl; print(ray.__version__, verl.__file__)'

再记录 resolved config 中的 nodes/GPUs、各并行 size、placement/resource pool、模型 dtype、max prompt/response length 和 memory utilization。只有“8 张卡”不足以复现资源拓扑。

通关检查 ​

选择你的 resolved config，画一张“role → WorkerGroup/rank → node/GPU → engine parallelism”表。再分别说明 reference 与 critic 为什么存在/省略、rollout server 与训练 actor 是否共置、一次权重同步跨过哪些进程边界。
