这一课讲什么
04 · 源码LESSON 14 40–55 分钟源码入门

这一课要回答：谁决定下一步做什么、谁真正占 GPU 计算、样本与权重又分别怎样流动？

整体架构：控制面、数据面与计算面 ​

当前 V1 可以用三层理解：Hydra/Ray driver 决定“做什么”，WorkerGroup 与 LLM server 执行“怎么算”，TransferQueue 保存“样本在哪里”。

先用人话：导演、演员与道具仓库 ​

  • 控制面像导演：排顺序、叫哪个角色行动，但自己通常不做大模型矩阵计算；
  • 计算面像演员：actor/critic/rollout/reward worker 真正占用 GPU/CPU；
  • 数据面像编号道具仓库：TransferQueue 保存值，trainer 常拿的是 KVBatchMeta 这张领料单；
  • 权重同步像换剧本：训练 actor 写出新版本，再交给推理 server 用于下一轮生成。

这个比喻只负责建立边界。接下来用真实组件和调用关系校准。

组件图 ​

图的关键不是框的数量，而是两条边界：生成结果由 AgentLoop 写入 TQ，trainer 主要拿 KVBatchMeta 句柄；权重更新发生在训练 WorkerGroup，随后由 checkpoint engine 同步到 LLM server replicas。

组件职责 ​

main_ppo.py 与 TaskRunner ​

Hydra 组合配置、校验角色需求、初始化 Ray，并让远端 TaskRunnerV1 运行。TaskRunner 强制启用 TransferQueue，选择 sync、colocate_async 或 separate_async trainer，最后负责关闭 TQ。

PPOTrainer ​

它是单控制器上的编排者，不是所有大张量的永久容器。它初始化 tokenizer/dataloader、资源池、训练 WorkerGroup、reward loop、LLM servers 和 checkpoint engine；在 fit() 中按 step 安排验证、rollout、训练、保存、指标与清理。

AgentLoop 与 LLM server ​

AgentLoop 面向“一个 prompt 怎样与模型/工具/环境交互”。单轮回答只是最简单情况；多轮 agent 可以产出同一 session 的多个 output。LLM server 提供高吞吐生成，AgentLoop 将输出整理成统一字段后写 TQ。

TransferQueue 与 ReplayBuffer ​

TQ 的 tag 存轻量状态和长度/版本元数据，value 存 trajectory 字段。ReplayBuffer 同步 tag，等待完整 prompt group，优先取较旧的可用 prompt，并对超过版本阈值的轨迹执行 drop 或 wait。

WorkerGroup ​

训练计算由 Ray WorkerGroup 分发到数据并行/张量并行等后端。Actor、reference 与 rollout 可以共享物理资源或模型载入方式；critic、reward model、teacher 是否出现取决于配置。

三种 trainer mode ​

模式资源关系参数同步适合先理解什么
synctrainer 与 rollout 共置每个 step 更新后同步最清晰的正确性主线
colocate_async共置但生成/训练流水化允许一定异步空闲时间与 staleness 权衡
separate_async训练和 rollout 分离parameter_sync_step 控制频率独立扩缩容与更强异步

异步不是单纯“更快”的开关：它改变样本来自哪个权重版本、缓存容量、反压位置和故障可见性。第一次读源码建议固定 sync，再对比两个异步子类重写的生命周期钩子。

一次数据与权重的闭环 ​

数据方向是顺时针，权重方向从训练 worker 回到生成 server。调试时先判断问题属于哪条流：样本没有到 TQ、字段写错、训练没有更新，还是新权重没有同步。

通关检查 ​

遮住组件图，分别画“数据箭头”和“权重箭头”。在图中圈出只做控制的对象、真正持有模型参数的进程、保存轨迹值的系统，以及只携带 keys/tags 的句柄。若把 PPOTrainer 画成亲自持有所有 GPU tensor，需要回看边界。
