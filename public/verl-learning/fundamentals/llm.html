LLM 与后训练：源码阅读所需的最小知识 ​

veRL 不负责从零解释 Transformer，但它大量操作 token、概率、mask 与缓存。本节只建立读懂 rollout 和训练接口所需的模型。

模型实际输出什么 ​

给定 token 前缀 x1:t，自回归模型输出词表上每个候选 token 的 logits。softmax 把 logits 变成概率：

pθ(xt+1∣x1:t)=softmax(zθ(x1:t)).

生成器再按 temperature、top-p、top-k 等规则采样。整段回答的对数概率是各 response token 条件对数概率之和；工程里经常保留逐 token log_prob，因为 mask、PPO ratio、KL 与长度归一化都需要它。

五个源码高频概念 ​

Tokenization ​

字符串先经过 tokenizer 变成整数 ID。prompt_length、response_length 指 token 数而不是字符数。中文、代码和空格的切分方式不同，所以不能用字符串长度替代 token 长度。

Attention mask 与 response mask ​

attention_mask 告诉模型哪些位置有效，通常覆盖 prompt 与 response；response_mask 决定哪些回答 token 参与奖励、优势和损失。padding、工具调用中的特殊区段或被过滤位置可能为 0。

一个稳妥的心智模型是：形状描述“内存里有多少位置”，mask 描述“统计上哪些位置算数”。

Log-probability 与 entropy ​

log_prob 是被选 token 的对数概率。熵衡量整张分布的不确定性：分布越尖锐，熵越低。RL 训练中熵可作为监控项或正则项，但“熵升高/降低”本身不等于效果变好，必须与 reward、KL、长度和任务成功率一起看。

KV cache ​

自回归解码时，历史 token 的 attention key/value 可以缓存，避免每一步重复计算整个前缀。KV cache 提升 rollout 吞吐，但消耗显存；训练反向传播使用的激活与推理缓存不是同一种生命周期。veRL 在训练与 rollout 共置时会显式唤醒、休眠引擎并管理缓存/权重。

Training 与 inference 的数值差异 ​

rollout 引擎可能采用不同 kernel、精度或并行方式，训练 worker 会重新计算 old_log_prob。因此源码中会同时出现：

  • rollout_log_probs：生成引擎采样时的行为策略概率；
  • old_log_probs：训练侧冻结旧策略重新计算的概率；
  • 当前 actor 在 mini-batch 更新时的 log-prob；
  • ref_log_prob：参考模型概率，用于 KL 约束。

这些量理论上相关，但不能仅凭名称假设逐位完全相等。异步或训练/推理实现不同还会引入 staleness 与数值偏差。

后训练在做什么 ​

SFT 告诉模型“像示范那样回答”；RL 告诉模型“哪些自己生成的行为更值得提高概率”。veRL 的主要职责落在后一个闭环：批量 rollout、评估、估计优势、分布式更新和同步权重。

带着这些问题进入源码 ​

  1. 当前字段是 prompt+response 全长，还是只有 response 长度？
  2. padding 在左侧还是右侧？有效区间由哪个 mask 给出？
  3. log-prob 来自 rollout、旧 actor、当前 actor 还是 reference？
  4. 数据在 CPU、训练 GPU、推理 GPU，还是外部存储？
  5. 这一步会建立梯度，还是纯推理？
